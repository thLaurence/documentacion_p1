Documentación Api Rest

Introducción

Preámbulo

Este documento fue creado en base al funcionamiento y estructura de la API REST, esto para generar una mejor comprensión del equipo desarrollador y que a futuro se pueda agregar o realizar cambios al código. Aquí quedaran representado los métodos request GET, POST, PUT y DELETE. Además, los códigos de estado HTTP.

El Funcionamiento principal de nuestra API REST será la comunicación a través de solicitudes HTTP para realizar distintas funciones estándar en la base de datos de Alpha Docere, estas funciones van desde crear, leer, actualizar y eliminar registros (Conocido como CRUD) de la tabla llamada “miembros”. Es decir utilizará una solicitud GET para recuperar el registro de uno o varios miembros, una solicitud POST para crear un nuevo miembro, una solicitud PUT para actualizarlos y una solicitud DELETE para suprimir o eliminar un miembro de la base de datos. Cada uno de estos métodos HTTP pueden ser utilizados en llamadas a la API.

La API fue desarrollada en código PHP y con conexión a base de datos MYSQL. También quedaran detalladas las rutas de cada método, los parámetros y json de las peticiones con sus respectivos mensajes de respuestas.

Glosario General

Para una mejor comprensión del lector, es necesario definir algunos conceptos e identificadores que serán referidos en la documentación.

Token de acceso: un identificador único de un usuario que solicita acceso a la API. Un usuario puede generar un token de acceso en el panel de control. Cuando se realiza una solicitud, podemos asociar el token proporcionado con el que tenemos almacenado, para autenticar e identificar al usuario solicitante y otorgar los permisos de edición.

API: un puente entre dos sistemas que normalmente no trabajarían juntos. Por ejemplo, sistemas que no tienen un lenguaje común con cual que comunicarse.

Encabezado HTTP: los encabezados HTTP son los pares nombre-valor que aparecen en los mensajes de solicitud y respuesta de los encabezados de mensajes del protocolo de transferencia de hipertexto (http).

JSON: un formato de intercambio de datos de lenguaje natural y ligero. Es el formato que se utiliza para enviar y recuperar la mayoría de los datos a través de la API.

Transacciones

Flujo de una Transacción

El flujo de una Transacción a través de la API RESTful se compone de los siguientes Requests:

  1. Se realiza un GET para listar todos los miembros a http://localhost/ApiDocere/miembros. Parámetros solicitados page y value.

  2. Se realiza un GET para listar un miembro a http://localhost/ApiDocere/miembros. Parámetros solicitados id y value.

  3. Se realiza un POST para crear un miembro a http://localhost/ApiDocere/miembros Se envia por body un JSON con todos los atributos del miembro incluyendo el token para si creación

  4. Se realiza un POST para crear un miembro a http://localhost/ApiDocere/miembros Se envia por body un JSON con todos los atributos del miembro incluyendo el token para si creación

Códigos de Respuesta y Mensajes de Error

status_200

“OK”. Se refiere a que la solicitud ha sido aceptada. Es decir, la solicitud fue aceptada y se ha obtenido un resultado. Este es un código de respuesta general a cualquier solicitud que se genere. En el caso de una solicitud GET el registro solicitado ya se encuentra en el cuerpo de la respuesta. En una solicitud PUT o DELETE la solicitud fue aceptada y el reemplazo o la eliminación del registro ya se encuentra en el cuerpo de la respuesta.

status_400

“BAD REQUEST”. Datos en un formato incorrecto o incompletos. La solicitud no ha sido válida, esto sucede cuando el servidor ha intentado procesar la solicitud pero existe algún dato o no existe un dato por lo tanto no es válido y se rechaza la solicitud.

status_401

“UNAUTHORIZED”. No se encuentra autorizado, este código de respuesta es enviado cuando se encuentra habilitada la seguridad en el servidor y falta la información de autorización en la solicitud.

status_405

“METHOD NOT ALLOWED”. Método no permitido, esto ocurre cuando el recurso de destino no admite el método HTTP solicitado, un ejemplo es cuando solamente permite operaciones de GET.

Conclusión

Con el objetivo de obtener un código fluido y acertado para los distintos requerimientos solicitados, nuestra API REST cumple con los estándares propuestos al inicio del trabajo, estableciendo una comunicación con el cliente y exponiendo los datos de la clase miembros, logrando así el funcionamiento correcto del CRUD. Permitiendo el fácil manejo para un buen mantenimiento a futuro, pues al seguir la estructura creada como base en la clase miembros nos permite entender de mejor manera el código. Además, facilita el desarrollo de la Api para la creación de las clases que serian necesarias incluir. Nuestros estándares mínimos fueron establecidos gracias a la creación de una API Prototipo al comienzo del proyecto con la cual se realizaron las distintas pruebas, evitando así distintos errores comunes en un futuro.
